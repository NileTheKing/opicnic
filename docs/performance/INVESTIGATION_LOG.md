# Opicnic 성능 최적화 수사 일지 (Investigation Log)

본 문서는 성능 최적화 과정에서 발생한 이슈와 가설 검증, 최종 해결 과정을 기록한 실무 일지임.

---

## 1. 500 VU 부하 시 시스템 지연 발생
- **현상**: 동시 접속자 500명 도달 시 응답 지연이 1.1s로 고정되며 최종적으로 프로세스 종료.
- **분석 도구**: k6 (http_req_duration 지표).
- **식별된 병목**: AsyncConfig의 FixedThreadPool(10). 가상 스레드 환경에서도 내부 비동기 태스크가 한정된 플랫폼 스레드 풀에 갇혀 지연 발생.

## 2. 가상 스레드 엔진 전환 (Step 1)
- **조치**: taskExecutor를 VirtualThreadPerTaskExecutor로 교체.
- **결과**: RPS 96 ➡️ 242 (2.5배 향상). 지연 시간은 여전히 1.1s 수준 유지.

## 3. 데이터 정합성 오류 수정 및 가설 기각
- **현상**: 실 DB 조회 추가 후 성공률 0% 급락.
- **분석**: 로그 확인 결과 Enum mismatch 버그 발견하여 수정.
- **가설**: DB 커넥션 부족을 의심해 10 ➡️ 50 확장했으나 지표 변화 없음. DB 병목 가설 기각.

## 4. 1KB 격리 실험을 통한 병목 구간 특정 (Step 2)
- **가설**: 파일 크기에 따른 I/O 부하가 지연의 원인일 것임.
- **실험**: 1MB 파일을 1KB 아주 작은 파일로 교체하여 부하 테스트 수행.
- **결과**: p95 지연 시간이 132ms로 급감. 범인이 '파일 I/O'임을 논리적으로 확정.

## 5. JFR 분석 및 물리적 증거 확보 (Step 3)
- **분석**: JFR(Java Flight Recorder) 정밀 프로파일링 가동.
- **증거**: jdk.ObjectAllocationSample 분석 결과 대규모 byte[] 복사와 톰캣의 디스크 쓰기 이벤트 포착.
- **결론**: 톰캣의 멀티파트 기본 임계치(10KB)를 넘는 파일이 디스크 I/O Wait를 유발함.

## 6. 최종 해결 및 리팩터링
- **최적화**: file-size-threshold(2MB) 설정 및 InputStream 릴레이 구조 도입으로 디스크 쓰기 제거.
- **리팩터링**: 가상 스레드 특성에 맞춰 invokeAll() 기반의 동기 스타일 병렬 처리로 전환하여 가독성 확보.
- **최종 성과**: p95 Latency 1,130ms ➡️ 238ms (약 78% 단축).
