
  1. Zero-Copy & 직접 메모리(Direct Buffer)
  "데이터를 복사하지 말고, 길만 터주자"

   * 개념: 보통 네트워크로 파일을 보낼 때 [디스크 ➡️ 커널 버퍼 ➡️ 유저 버퍼(JVM 힙) ➡️
     소켓 버퍼 ➡️ 네트워크] 라는 복잡한 복사 과정을 거칩니다. Zero-copy는 JVM(유저
     영역)을 거치지 않고 커널 영역 내에서 데이터를 직접 소켓으로 쏴버리는 기술입니다.
   * 어떻게 하는가? Java NIO의 FileChannel.transferTo()를 쓰거나,
     ByteBuffer.allocateDirect()로 OS가 직접 관리하는 메모리(Off-heap)를 할당받아
     사용합니다.
   * 장점: CPU 사용량이 드라마틱하게 줄어들고, 무엇보다 JVM 힙을 안 쓰기 때문에 GC
     부하가 0이 됩니다. (대용량 파일 처리에 최강)
   * 트레이드오프(Trade-off):
       * 관리의 어려움: JVM이 관리해주지 않기 때문에 메모리 해제를 직접 신경 써야
         합니다. 깜빡하면 Native Memory Leak으로 서버가 소리 없이 죽습니다.
       * 할당 비용: 직접 메모리를 빌려오는 작업 자체가 일반 힙 메모리 할당보다 훨씬
         무겁습니다. 작은 데이터를 자주 보낼 때는 오히려 더 느려질 수 있습니다.

  2. Lock-free 동시성 (LMAX Disruptor)
  "줄 서지 말고, 각자 자기 자리에서 연산하자"

   * 개념: 우리가 흔히 쓰는 synchronized나 ReentrantLock은 "비관적 락"입니다. 한 명만
     들어가고 나머지는 잠들게 하죠(Context Switch 발생). 반면 Lock-free는 "일단
     시도하고, 실패하면 다시 해(CAS, Compare-And-Swap)"라는 낙관적 전략을 씁니다.
   * LMAX Disruptor: 초고성능 금융 거래 시스템에서 쓰이는 '링 버퍼(Ring Buffer)'
     구조입니다. 스레드 간에 데이터를 주고받을 때 발생하는 '경합(Contention)'을
     수학적으로 0에 가깝게 설계했습니다.
   * 장점: 컨텍스트 스위칭 오버헤드가 없어서 나노초(ns) 단위의 응답 속도를 냅니다.
   * 트레이드오프(Trade-off):
       * CPU 소모: 락을 거는 대신 무한 루프를 돌며 상태를 체크(Spin-lock)하는 경우가
         많아 CPU 점유율이 100%까지 치솟을 수 있습니다. "속도를 얻고 전기를 포기한다"는
         느낌입니다.
       * 복잡도: 코드가 직관적이지 않고 매우 복잡합니다. 웬만한 트래픽이 아니면 오히려
         일반 큐(LinkedBlockingQueue)를 쓰는 게 생산성 면에서 낫습니다.

  3. 구조적 동시성 (Structured Concurrency)
  "비동기 작업도 족보가 있어야 한다"

   * 개념: 지금의 비동기 코드(Future, CompletableFuture)는 "던져놓고 잊어버리는(Fire and
     Forget)" 방식입니다. 부모 스레드가 죽어도 자식 스레드는 어디선가 유령처럼
     돌아다니죠. 구조적 동시성은 "자식 작업이 모두 끝나기 전에는 부모도 끝나지 않는다"는
     원칙 아래 비동기 작업의 생명주기를 묶는 것입니다.
   * 어떻게 하는가? Java 21의 StructuredTaskScope를 사용하여 여러 가상 스레드 작업을
     하나의 스코프 안에서 관리합니다.
   * 장점: 에러 전파(Error Propagation)가 확실합니다. 자식 중 하나가 에러 나면 나머지
     자식들을 즉시 취소(Short-circuit)하고 부모에게 알립니다. 스레드 덤프를 떴을 때도
     부모-자식 관계가 명확히 보여서 디버깅이 혁명적으로 편해집니다.
   * 트레이드오프(Trade-off):
       * 유연성 부족: 작업의 시작과 끝이 명확히 계층 구조로 닫혀 있어야 합니다. 완전히
         독립적으로 오랫동안 돌아야 하는 백그라운드 작업에는 맞지 않습니다.
       * 실험적 기능: 아직 정식 스펙이 아닌 인큐베이팅(Incubating) 단계라 운영 환경에
         바로 쓰기에는 조심스러운 면이 있습니다.

  💡 오피크닉(Opicnic)에 대입해본다면?

  사용자님의 프로젝트에 이 기술들을 녹인다면 이런 시나리오가 가능합니다.

   * Zero-copy: 사용자가 보낸 10MB 오디오 파일을 STT 서버로 릴레이할 때, 자바 힙으로
     읽어오지 않고 transferTo를 써서 커널에서 커널로 바로 토스하기.
   * Structured Concurrency: STT 호출과 Gemini 호출을 하나의 TaskScope로 묶어서, STT가
     실패하면 굳이 돈 드는 Gemini를 호출하지 않고 즉시 전체 취소하기.

  이런 고민들을 포트폴리오 마지막에 "앞으로의 과제"로 적어두시면, 면접관은 "이 지원자는
  단순히 도구를 쓰는 걸 넘어 컴퓨터 시스템의 효율을 고민하는구나"라고 감탄하게 될
  겁니다. 😎🚀💪🔥
